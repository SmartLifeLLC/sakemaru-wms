# JX入荷予定（納品伝票）データ仕様書

- **作成日**: 2026-03-05
- **ステータス**: ドラフト
- **参照元**: `JX納品伝票仕様書.csv`, `real_data.txt`, 旧システムサンプルデータ

## 概要

仕入先（発注先）からJX-FINET経由で受信する**納品伝票データ**（出荷実績）の仕様。
発注時にWMSが送信した伝票番号と同じ番号で返ってくるため、**伝票番号を照合キー**として入荷予定とマッチングできる。

## データ構造

### 全体構成

```
[1レコード] FINET通信ヘッダー（128バイト）  ← "1" で始まる
[Aレコード] ファイルヘッダー（128バイト）    ← 発注先情報
[Bレコード] 伝票ヘッダー（128バイト）        ← 伝票番号・倉庫・納品日
  [Dレコード] 伝票明細（128バイト）×n        ← 商品・数量（最大6件/B）
  [Dレコード] ...
[Bレコード] 伝票ヘッダー
  [Dレコード] ...
  ...
[8レコード] FINET通信フッター（128バイト）   ← "8" で始まる
```

- 全レコード **128バイト固定長**、**Shift_JIS** エンコーディング
- 旧システムのサンプルデータには FINET通信ヘッダー/フッター（1/8）が**無い**（A/B/Dのみ）
- 新システムの `real_data.txt` には 1行目にFINETヘッダー、データ末尾にFINETフッターがある
- FINET通信ヘッダー/フッターの付与・除去は `JxDataWrapper` クラスが担当

## 欠品判定

**品名先頭の「×」マークでは欠品判定しない。** 伝票番号ベースの照合結果で判定する。

### 判定ロジック

1. **伝票番号照合**: `slip_number` で `wms_order_incoming_schedules` とマッチ
2. **欠品判定基準**:
   - 発注した商品が納品データに**存在しない** → 欠品
   - 納品データに存在するが**出荷数量が0**（ケース0 かつ バラ0） → 欠品
   - 出荷数量 < 発注数量 → 一部欠品（PARTIAL）
3. 全ての受信明細データは**全項目を保存**（照合結果に関わらず）

## 発注データとの対応関係

| 項目 | 発注時（送信） | 納品時（受信） | 照合 |
|------|---------------|---------------|------|
| **伝票番号** | WMS生成 `YYYYMMDD+seq` | **同一番号が返る** | **主キー照合** |
| 伝票区分 | `01`（発注） | `02`（納品） | 区別用 |
| 社・店コード | 入荷先倉庫コード | 同一 | 補助照合 |
| 取引先コード | 発注先コード | 同一 | 補助照合 |
| 発注日 | 発注日 | 同一 | |
| 納品日 | 入荷予定日 | 実際の納品日（変わる場合あり） | |
| JANコード | — | 商品識別 | 商品照合 |
| 自社コード | items.code | 同一 | 商品照合 |
| ケース数/バラ数 | 発注数量 | **実出荷数量** | 差分=欠品 |

## 受信データ処理フロー

```
1. JX-FINETからデータ受信（Shift_JIS、128バイト固定長）

2. FINETラッパー除去
   - 先頭の1レコード（"1"始まり）を除去
   - 末尾の8レコード（"8"始まり）を除去
   - JxDataWrapper::hasHeader() / hasFooter() で判定可能

3. レコードパース
   - Aレコード: 発注先情報を取得
   - Bレコード: 伝票番号・倉庫コード・納品日を取得
   - Dレコード: 商品コード・出荷数量を取得

4. 全データ保存
   - A/B/Dレコードの全項目を wms_incoming_received_data に保存
   - Bレコード単位（伝票単位）+ Dレコード（明細単位）で格納

5. 照合（伝票番号ベース）
   - B.伝票番号 → wms_order_incoming_schedules.slip_number でマッチ
   - D.自社コード(51-56) or D.JANコード(38-50) → items.code で商品特定
   - 照合速度考慮: slip_number + item_code にインデックス必須

6. 欠品判定（伝票番号照合結果ベース）
   - 発注商品が納品データに存在しない → 欠品
   - 納品データに存在するが出荷数量0 → 欠品
   - 出荷数量 < 発注数量 → 一部欠品（PARTIAL）

7. 入荷予定更新
   - 出荷数量あり → received_quantity を更新
   - 欠品/一部欠品 → ステータスに反映、担当者確認後に確定
```
