# JX発注データ vs 入荷予定データ 比較分析

- **作成日**: 2026-03-05
- **ステータス**: ドラフト
- **ディレクトリ**: `storage/specifications/incoming/入荷予定対応/20260305-incoming-shipment-data-integration/incoming-samples/jx-data/`

## 背景・目的

WMSが送信した発注データ（`1330_order_20260303124418.txt`）に対して、発注先から返ってくる入荷予定データ（納品伝票）の構造を比較し、現在の`JxIncomingParser`が**伝票番号照合**と**商品コード変換**を正しく行えるか分析する。

## 1. レコードレイアウト比較

### 1.1 全体構造

| 項目 | 送信データ（発注） | 受信データ（納品） |
|------|-------------------|-------------------|
| FINETヘッダー（1レコード） | あり（data_type=`91`） | あり（data_type=`90`） |
| Aレコード | data_type=`01` | data_type=`02` |
| Bレコード | slip_type=`01`（発注） | slip_type=`02`（納品） |
| Dレコード | **品名64バイト** | **品名64バイト（実データ）** |
| FINETフッター（8レコード） | あり | あり |

### 1.2 Dレコード レイアウト — 仕様書 vs 実データ

### 1.3 実データ検証結果

**旧サンプル（FINETなし）** と **新real_data.txt（FINETあり）** の両方で、Dレコードは**送信レイアウト（品名64バイト）**を使用していることを確認:

```
real_data.txt D record例:
  Product=[×ハイネケン・瓶３３０ｍｌ] JAN=[4901411147017] Own=[142064] Pack=[000030] Cases=[0000000] Pieces=[0000000]
```

仕様書の32バイト位置でパースするとJAN・自社コードが**文字化け**する。

### 1.4 パーサーの現状

`JxIncomingParser`（L209-212）は**送信レイアウト（品名64バイト）** を使用 → **正しい**。

```php
// 実データは送信レイアウトと同一（品名64バイト）
// 6-69:product_name(64) 70-82:jan_code(13) 83-88:item_code(6) ...
```

## 2. 伝票番号照合 — **不一致あり（要修正）**

### 2.1 フォーマットの違い

| 段階 | フォーマット | 例 | 桁数 |
|------|------------|-----|------|
| DB保存（`wms_order_incoming_schedules.slip_number`） | `YYYYMMDD-NNNNN` | `20260303-00001` | 14文字 |
| JX送信（Bレコード position 4-14） | `YYYYMMDDNNN` | `20260303001` | **11文字** |
| パーサー変換後 | `YYYYMMDD-NNN` | `20260303-001` | **12文字** |

### 2.2 問題1: 送信時の伝票番号切り詰め

`HanaOrderJXFileGenerator.generateBRecord()` L346-350:

```php
// DB値はハイフン付き（20260305-00001）なのでハイフンを除去して11桁にする
if ($slipNumber) {
    $slipNumber = str_replace('-', '', $slipNumber);
    // → "2026030300001" = 13文字 → 11バイトフィールドに収まらない！
}
```

- DB伝票番号: `20260303-00001` → ハイフン除去 → `2026030300001`（**13文字**）
- Bレコードのフィールド幅: **11バイト**
- 結果: フィールドオーバーフロー（後続フィールドが2バイトずれる）

**現状**: send file では DB伝票番号が見つからず fallback が使われている（`Ymd + seq3` = 11文字）ため、まだ顕在化していない。

### 2.3 問題2: パーサーの変換不一致

`JxIncomingParser.formatSlipNumber()` L282-294:

```php
// "20260303001" → "20260303-001"  (3桁連番)
return substr($raw, 0, 8) . '-' . substr($raw, 8);
```

照合時: `20260303-001` ≠ `20260303-00001` → **マッチしない**

### 2.4 送信ファイルの実データ

```
B record: slip_no=[20260303001] → parser変換 → "20260303-001"
DB schedule:                                    "20260303-00001"
                                                ↑ 不一致！
```

### 2.5 修正案

**案A（推奨）**: 送信時にDB伝票番号を11桁に変換するロジックを統一

```php
// HanaOrderJXFileGenerator: DB伝票番号 → JX 11桁
private function toJxSlipNumber(string $dbSlipNumber): string
{
    // "20260303-00001" → date部分(8) + seq部分を3桁に変換
    $parts = explode('-', $dbSlipNumber);
    $date = $parts[0];              // "20260303"
    $seq = (int)($parts[1] ?? 0);   // 1
    return $date . str_pad($seq, 3, '0', STR_PAD_LEFT); // "20260303001"
}

// JxIncomingParser: JX 11桁 → DB伝票番号
private function formatSlipNumber(string $raw): string
{
    // "20260303001" → "20260303-00001" (5桁ゼロ埋め)
    $date = substr($raw, 0, 8);
    $seq = (int)substr($raw, 8);
    return $date . '-' . str_pad($seq, 5, '0', STR_PAD_LEFT);
}
```

**注意**: seq が 999 を超える場合（1日に1000件以上の伝票）は11桁に収まらない。運用上は問題ないか要確認。

## 3. 商品コード変換 — **動作確認OK（ただし注意点あり）**

### 3.1 現在の照合フロー

```
受信Dレコード → d_item_code (position 83-88) → items.code で照合
```

### 3.2 送信ファイルの商品コード

```
D record例:
  JAN=[4973480311460] OwnCode=[151242] ← items.code
  JAN=[4901411035123] OwnCode=[184535] ← items.code
```

`OwnCode`（自社コード）は `items.code` と一致するため、照合可能。

### 3.3 受信データ（real_data.txt）の商品コード

```
D record例:
  JAN=[4901411147017] Own=[142064] ← items.code
  JAN=[4902701908714] Own=[242063] ← items.code
```

同じく `items.code` と一致。

### 3.4 注意点

1. **自社コードが6桁固定**: items.code が6桁未満の場合、左ゼロ埋めされている可能性あり。照合時に `ltrim('0')` が必要かもしれない。
2. **JAN コード未使用**: 現在のパーサーは `d_jan_code` を保存するが照合には使っていない。自社コードで解決できない場合のフォールバックとして `item_search_information.search_string` でJAN照合を追加する余地あり。
3. **受信データの発注先側商品コード**: 仕様書の position 51-56（旧レイアウト）は発注先独自の商品コード。実データでは position 83-88（送信レイアウト）が `items.code`。パーサーは正しい位置を使用。

## 4. Aレコード パース — **動作問題なし**

### 4.1 送信 vs 受信 Aレコード

| 項目 | 送信 | 受信（実データ） |
|------|------|----------------|
| data_type | `01` | `02` |
| 日付 | `20260303`（YYYYMMDD pos 4-11） | `20260303`（※仕様は YYMMDD pos 6-11 だが実データは YYYYMMDD） |

パーサーは送信レイアウトで A レコードをパースしており、実データでも正しく動作する。

## 5. まとめ

| 項目 | 状態 | 詳細 |
|------|------|------|
| Dレコード レイアウト | **OK** | 実データは送信レイアウト。パーサーも送信レイアウト使用 |
| Aレコード パース | **OK** | 送信レイアウトで正しくパースできる |
| 商品コード変換 | **OK** | `d_item_code` = `items.code` で照合可能 |
| **伝票番号 送信** | **要修正** | DB伝票番号(13文字)がBレコード(11バイト)にオーバーフローする可能性 |
| **伝票番号 照合** | **要修正** | パーサー変換 `20260303-001` ≠ DB `20260303-00001` で不一致 |

### 必要な修正

1. **`HanaOrderJXFileGenerator.generateBRecord()`**: DB伝票番号を11桁JXフォーマットに正しく変換
2. **`JxIncomingParser.formatSlipNumber()`**: JX 11桁を5桁連番のDBフォーマット（`YYYYMMDD-NNNNN`）に変換
3. **`jx-incoming-data-specification.md`**: Dレコードの位置定義を実データに合わせて修正（品名64バイト=送信レイアウト）

## 確認事項

1. **1日の伝票数上限**: JX Bレコードの連番は3桁（最大999）。1日に1000件以上の伝票がある場合、11桁に収まらない。運用上この制約は許容可能か？
2. **旧伝票番号の照合**: 旧システムの伝票番号（例: `91451025874`）は日付ベースではない。これらの照合はスキップで良いか？
3. **仕様書の位置定義**: `jx-incoming-data-specification.md` の N型32桁は「32文字×2バイト=64バイト」の意味か、それとも仕様書自体が別レイアウトを定義しているか？
