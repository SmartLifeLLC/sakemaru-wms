## 調査結果（2026-03-07 実施）

### 発注関連

#### 1. 発注データの自動生成と発注データの自動送信はきちんと区別されているか？

**→ はい。完全に分離されている。**

| 機能 | コマンド | ジョブ | サービス |
|------|---------|--------|---------|
| 自動生成 | `wms:auto-order-scheduled` | `ProcessOrderCandidateGenerationJob` | `OrderCandidateCalculationService` |
| 自動送信 | `wms:auto-order-transmit` | `ProcessAutoSendJob` | `OrderTransmissionService` |

送信側は「承認→確定→JXファイル生成→送信→精算ステータス更新」の7ステップで、生成フェーズとは完全に独立。

#### 2. 自動生成と自動送信は時間は別々に区別、管理されているか？

**→ はい。`wms_contractor_settings`で発注先ごとに独立管理。**

| 設定項目 | カラム | 用途 |
|---------|--------|------|
| 生成時刻 | `auto_order_generation_time` | 発注候補生成のトリガー時刻 |
| 送信時刻 | `transmission_time` | 自動送信のトリガー時刻 |
| 曜日制御 | `is_transmission_{mon〜sun}` | 曜日ごとの送信可否 |
| 自動送信フラグ | `is_auto_transmission` | 自動送信の有効/無効 |

両コマンドとも`routes/console.php`で5分間隔ポーリング、`onOneServer()->withoutOverlapping()`で重複実行防止済み。

#### 3. 手動生成した発注データ競合は発生せずまとまるのか？

**→ はい。設計上、競合は発生しない。**

- 手動発注は`OrderExecutionService::createManualIncomingSchedule()`で直接`wms_order_incoming_schedules`に`order_source=MANUAL`で作成
- 自動送信ジョブは`wms_order_candidates`テーブルのPENDING/APPROVED/CONFIRMEDのみ処理
- 手動入庫予定は候補テーブルをバイパスするため、自動送信と干渉しない
- UIでは同一テーブルに表示されるが、ソースラベル（発注/手動/移動/受信）で明確に区別

#### 4. 自動生成された発注データと手動生成は紐付けができているか？

**→ 部分的。構造上は分離設計。**

| フィールド | 自動生成 | 手動生成 |
|-----------|---------|---------|
| `order_source` | `AUTO` | `MANUAL` |
| `order_candidate_id` | FK → `wms_order_candidates.id` | `NULL` |
| `manual_order_number` | `NULL` | ユーザー入力の注文番号 |

自動発注は`WmsOrderCandidate → WmsOrderIncomingSchedule`のリレーションで追跡可能。手動発注は候補テーブルへの逆参照はなし（設計上不要）。

#### 5. 自動生成用のコマンド・自動送信用のコマンドは用意されているか？

**→ はい。スケジュール用・手動実行用の両方が用意されている。**

**スケジュール実行用:**
| コマンド | 用途 |
|---------|------|
| `wms:auto-order-scheduled` | 定期発注候補生成（ジョブディスパッチ） |
| `wms:auto-order-transmit` | 定期自動送信（ジョブディスパッチ） |

**手動実行用:**
| コマンド | 用途 |
|---------|------|
| `wms:auto-order-calculate [--skip-snapshot] [--force]` | 手動で発注候補計算を実行 |
| `wms:transmit-orders [--batch-code=] [--dry-run]` | 手動で送信（ドライラン対応） |
| `wms:snapshot-stocks` | 在庫スナップショットのみ実行 |

#### 6. 自動生成、自動送信のスケジュリングはできているか？

**→ はい。`routes/console.php`に設定済み。**

```
wms:auto-order-scheduled  → 5分間隔 → logs/auto-order-scheduled.log
wms:auto-order-transmit   → 5分間隔 → logs/auto-order-transmit.log
```

同日の重複実行は`WmsAutoOrderExecutionLog`で`hasRunOrSucceededToday()`チェックにより防止。

### 入荷関連

#### 7. 入荷予定データは発注データに基づいて正しく生成されているか？

**→ はい。`OrderExecutionService::createIncomingSchedulesFromCandidate()`で実装済み。**

- `demand_breakdown`あり → 倉庫ごとに分割して入庫予定作成（マルチ倉庫対応）
- `demand_breakdown`なし → 1レコード作成（`expected_quantity = candidate->order_quantity`）
- 移動の場合は`TransferCandidateExecutionService`が別経路で処理

#### 8. 入荷予定データは発注数と入荷予定数、入荷確定数はそれぞれ別々に管理しているか？

**→ 部分的。構造上の注意点あり。**

| カラム | 用途 | 備考 |
|-------|------|------|
| `expected_quantity` | 入荷予定数 | 候補の`order_quantity`からコピー |
| `received_quantity` | 入荷確定数 | 初期値0、確定時に更新 |
| `shortage_quantity` | 欠品数 | 受信データ照合時に設定 |

**注意:** 元の「発注数」（`WmsOrderCandidate.order_quantity`）は入庫予定テーブル上では`expected_quantity`にコピーされるのみで、別カラムとしては保持されていない。API連携時も`expected_quantity`のみ公開。ただし、`order_candidate_id`経由で元の候補レコードの`order_quantity`は参照可能。

#### 9. 入荷伝票番号の管理はされているか？

**→ はい。実装済み。**

- `WmsOrderIncomingSchedule::generateSlipNumber()`: `{YYYYMMDD}{3桁連番}` = 11桁数字
- `slip_number`カラムにUNIQUE制約あり
- JX Bレコード送信時に11桁でそのまま格納
- 受信時に同じ伝票番号で照合（ラウンドトリップ）

#### 10. 発注先別に出荷実績データの受信時刻設定はできるようになっているか？

**→ はい。`wms_contractor_settings`で発注先ごとに設定可能。**

| 設定項目 | カラム |
|---------|--------|
| 受信有効化 | `is_receive_enabled` |
| 受信フォーマット | `receive_format`（JX/CSV） |
| 受信時刻 | `receive_time` |
| 曜日制御 | `is_receive_sun`〜`is_receive_sat` |

`wms:incoming-receive-scheduled`コマンドは`is_receive_enabled=true`かつ`receive_time`設定済みの発注先のみ処理。同日の重複受信チェックも実装済み。

#### 11. 出荷実績データを受け取った際に、既存の入荷予定データとマッピングできるようになっているか？

**→ はい。2つの入口で対応。**

| 入口 | 方法 | 自動照合 |
|------|------|---------|
| JXデータサーバ受信 | `wms:incoming-receive-scheduled` → `IncomingReceiveService::parseJxData()` → `matchWithSchedules()` | あり |
| ウェブファイルアップロード | 管理画面「JXデータ取込」ボタン → `IncomingReceiveService::parseJxData()` | パースのみ（照合は別操作） |

パーサー（`JxIncomingParser`）は3層テーブルに保存:
- `wms_incoming_received_files`（Aレコード）
- `wms_incoming_received_slips`（Bレコード）
- `wms_incoming_received_details`（Dレコード）

**注意:** CSV形式は設定上は存在するが、実装は未対応（コマンド内で「CSV形式は未対応」と明記）。

#### 12. 出荷実績データにレコードがないものは入荷予定0として更新されるか？

**→ 部分的に対応。**

照合ロジック（`IncomingReceiveService`）:
- 伝票番号でマッチした後、Dレコードの`d_item_code`に該当商品がない場合 → `SHORTAGE`ステータス
- `shortage_quantity = expected_quantity`（全数欠品）として記録
- `received_quantity`は0のまま（暗黙的に0、明示的な0書き込みはなし）
- `is_receive_matched = true`で照合済みフラグを設定

結果としてセマンティクスは正しい（受信数=0、欠品数=発注数）が、`received_quantity`への明示的な0書き込みは行っていない。

#### 13. 出荷実績データにあって入荷予定にないものは新規生成されるか？区別できるか？

**→ はい。両方対応済み。**

`IncomingReceiveService::createSchedulesFromSlip()`で:
- 伝票番号で既存入庫予定が見つからない場合 → 新規`WmsOrderIncomingSchedule`を作成
- `order_source = OrderSource::RECEIVED`で区別
- 伝票の`match_status = 'CREATED'`（既存マッチの場合は`'MATCHED'`）

UI上のラベル表示:
- `AUTO` → 「発注」（青）
- `MANUAL` → 「手動」（グレー）
- `TRANSFER` → 「移動」
- `RECEIVED` → 「受信」（緑）

#### 14. 入荷予定データのAPI連携時に、発注数・予定数がそれぞれ連携されるか？

**→ いいえ。`expected_quantity`のみ連携。**

ハンディターミナルAPI（`IncomingController::formatScheduleDetail()`）のレスポンスには:
- `expected_quantity`（予定数）
- `received_quantity`（確定数）
- `remaining_quantity`（残数 = 予定数 - 確定数）

元の「発注数」（`WmsOrderCandidate.order_quantity`）はAPI上に別フィールドとして公開されていない。
作成時に`expected_quantity = order_quantity`でコピーされるため、初期値は同一。

#### 15. 入荷確定が完了したら仕入データ生成依頼が生成されるか？

**→ はい。ただし自動ではなく手動トリガー。**

**ステップ1:** `IncomingConfirmationService::confirmIncoming()` → ステータスを`CONFIRMED`に更新（仕入連携はこの時点では行わない）

**ステップ2:** 管理画面「入荷済」タブの「仕入データ登録」ボタン → `IncomingTransmissionService::transmitConfirmedIncomings()`:
1. `CONFIRMED`ステータスの入庫予定を取得
2. 倉庫コード + 仕入先コード + 入荷日でグルーピング
3. 100件ごとにチャンク分割
4. `purchase_create_queue`テーブルにINSERT（sakemaruデータベース）
5. 入庫予定のステータスを`TRANSMITTED`に更新

**注意:** 入荷確定 → 仕入データ生成は**自動ではない**。管理者が手動で「仕入データ登録」ボタンを押す必要がある。

### C. 入荷予定への単価カラム追加

**要件:** `wms_order_incoming_schedules`に以下を追加:

| カラム | 用途 |
|-------|------|
| `unit_price` | 仕入自社単価 |
| `partner_unit_price` | 仕入先単価 |

**自社単価（`unit_price`）の取得方法:**

core側コード `Item::currentPartnerPriceVO()` をDB直接参照で再現する（WMS側にcoreコードの依存は持たない）。

```
Step 1: 仕入先別単価 — item_partner_prices (partner_id, warehouse_id)
Step 2: 個別単価G — item_partner_prices (partner_price_group_id)
Step 3: 個別単価G2 — item_partner_prices (partner_price_group2_id)
Step 4: 商品単価 — item_prices → items.purchase_price_type で分岐
         PRODUCER → producer_unit_price
         COST     → cost_unit_price
         WHOLESALE → wholesale_unit_price
```

各ステップのSQL:
```sql
-- Step 1-3: item_partner_prices から取得（partner_idを差し替えて3回検索）
SELECT * FROM item_partner_prices
WHERE item_id = :item_id
  AND partner_id = :partner_id  -- Step1: supplier, Step2: partner_price_group_id, Step3: partner_price_group2_id
  AND (warehouse_id = :warehouse_id OR warehouse_id IS NULL)
  AND start_date <= :process_date
ORDER BY start_date DESC LIMIT 1;

-- Step 4: 商品単価（上記すべてNULLの場合）
SELECT ip.*, i.purchase_price_type
FROM item_prices ip
JOIN items i ON i.id = ip.item_id
WHERE ip.item_id = :item_id
  AND ip.start_date <= :process_date
ORDER BY ip.start_date DESC LIMIT 1;
-- → purchase_price_type に応じて producer/cost/wholesale の単価カラムを参照
```

**仕入先単価（`partner_unit_price`）の取得方法:**

| データソース | `partner_unit_price`のマッピング元 | 備考 |
|------------|----------------------------------|------|
| JX納品伝票 | Dレコード原単価 [109-118]（10桁、下2桁小数） | 単価種別は出荷形態で判定（下記参照） |
| アクト中食CSV | `売価単価`カラム | |

**JX Dレコード原単価の解釈:**

原単価は10桁で下2桁が小数点（例: `0002352 00` → 2352.00円）。
単価がケース単価かバラ単価かは**出荷形態で判定**:

| ケース数 | バラ数 | 単価種別 |
|---------|--------|---------|
| > 0 | any | ケース単価 |
| 0 | > 0 | バラ単価 |
| 0 | 0（欠品） | ケース単価（デフォルト） |

**JX Dレコード実データ例:**
```
位置70-128: 4902102069359 211222 000024 0000001 0000000 0002352 00
            |JAN(13)     |code(6)|入数(6)|CS(7)  |バラ(7) |単価(10)   |
```
この場合: CS=1, バラ=0 → ケース単価 2352.00円

**送料の特別扱い:**
- JAN `9999999999996` = 送料。仕入先単価をそのまま採用し、ワーニング・エラー対象外とする
- アクト中食ではこの商品コードは相手単価利用として固定対応

**運用:**
- 自動発注・手動発注の生成時: DBから`unit_price`を取得し、`partner_unit_price = unit_price`として登録
- 出荷実績データ取込時: `partner_unit_price`を取込データの単価で更新
- 単価不一致（`unit_price != partner_unit_price`）時: 出荷実績エラーリストにワーニング表示
- 例外: JAN `9999999999996`（送料）は単価比較対象外

**関連テーブル（すべてsakemaruDB、core側）:**

| テーブル | 役割 |
|---------|------|
| `item_partner_prices` | 仕入先別・倉庫別の個別単価 |
| `item_prices` | 商品マスタ単価（生販/仕入/卸/売上/小売等） |
| `items` | `purchase_price_type`で仕入単価区分を保持 |
| `partners` | `partner_price_group_id`, `partner_price_group2_id`で親グループ参照 |

### E. 入荷確定→仕入データ生成フローの詳細

**Q: 入荷確定が手動であるという意味か？確定を押すと仕入れデータは生成ロジックに入るか？**

**→ 入荷確定も仕入データ生成も、どちらも手動トリガー。2段階の手動操作が必要。**

#### ステップ1: 入荷確定（手動）

2つの入口:

| 入口 | トリガー | 処理 |
|------|---------|------|
| ハンディターミナル | `POST /api/incoming/work-items/{id}/complete` | `IncomingConfirmationService::confirmIncoming()` |
| 管理画面 | 「入庫確定」ボタン / 「選択を入庫確定」一括アクション | 同上 |

`confirmIncoming()`の処理内容:
1. ステータス検証（CONFIRMED/CANCELLED済みは拒否）
2. `received_quantity` 更新（未指定時は`expected_quantity`をコピー）
3. `actual_arrival_date` 更新（未指定時は当日）
4. ステータス → `CONFIRMED`
5. `confirmed_at` / `confirmed_by` 記録
6. 移動（TRANSFER）の場合のみ `stock_transfer_queue` に自動INSERT

**重要: この時点では仕入データは一切生成されない。**

#### ステップ2: 仕入データ生成（手動）

- 管理画面「入荷済」タブの「仕入データ登録」ボタン（`ListWmsIncomingCompleted.php`）
- `IncomingTransmissionService::transmitConfirmedIncomings()` を呼び出し
- `CONFIRMED`のレコードを`purchase_create_queue`にINSERT → ステータスを`TRANSMITTED`に更新
- ボタンは`CONFIRMED`レコードが存在する場合のみ表示

**自動トリガーは一切なし:** イベント、リスナー、オブザーバー、スケジュールジョブのいずれも未設定。
