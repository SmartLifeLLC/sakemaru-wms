# 入荷検品 API仕様（サーバ側・端末側）

## 概要

入荷検品は3つの数量を段階的に管理する。

```
発注 → 出荷 → 検品（入庫）
```

| 段階 | カラム | 意味 | 設定タイミング |
|------|--------|------|--------------|
| 発注 | `expected_quantity` | 発注数（入荷予定数） | 発注確定時 |
| 出荷 | `shipped_quantity` | 出荷実績数（仕入先からの出荷数） | JX/CSV受信・照合時 |
| 検品 | `received_quantity` | 入庫確定数（検品で確認した実数） | Handy検品完了時 or 管理画面確定時 |

補助:
| カラム | 意味 |
|--------|------|
| `shortage_quantity` | 欠品数（`expected - shipped`） |
| `is_receive_matched` | 出荷実績データと照合済みか |

---

## サーバ側: 数量管理ルール

### ルール1: 各数量は独立して管理する

`received_quantity` は物理的な検品結果のみで更新される。
JX/CSV照合で `shipped_quantity` を設定しても `received_quantity` は変更しない。

```
× JX照合時に received_quantity = shipped_quantity とする（旧動作）
○ JX照合時は shipped_quantity のみ設定。received_quantity は検品で設定
```

### ルール2: デフォルト検品数量の決定

端末で検品作業を開始する際のデフォルト数量:

```
if shipped_quantity が存在する（> 0）:
    デフォルト = shipped_quantity - received_quantity  ← 出荷実績ベース
else:
    デフォルト = expected_quantity - received_quantity  ← 発注数ベース
```

**理由**: 出荷実績があれば、実際に届く数量はその数。発注数より正確。

### ルール3: 検品入力は常にデフォルトより優先

端末で作業者が数量を変更した場合、その入力値が最終的な `received_quantity` になる。
デフォルト値はあくまで入力支援。

### ルール4: 入庫完了判定

```
if received_quantity >= expected_quantity:
    status = CONFIRMED（入庫完了）
elif received_quantity > 0:
    status = PARTIAL（一部入庫）
else:
    status = PENDING（未入庫）
```

### ルール5: `received_quantity` の更新経路

| 経路 | 処理 | 備考 |
|------|------|------|
| Handy検品完了 | `POST /api/incoming/work-items/{id}/complete` | 作業者の物理カウント |
| 管理画面入庫確定 | `IncomingConfirmationService::confirmIncoming()` | 管理者操作 |
| 出荷実績適用 | `IncomingReceiveService::applyMatched()` | 管理者が「適用」ボタンを押した場合のみ |

**JX照合（`matchWithSchedules`）では `received_quantity` を更新しない。**

## サーバ側: データフロー

### パターンA: 出荷実績あり → Handy検品

```
1. 発注確定
   expected_quantity = 10
   shipped_quantity  = null
   received_quantity = 0
   status = PENDING

2. JX受信・照合
   shipped_quantity  = 8
   shortage_quantity = 2
   is_receive_matched = true
   ※ received_quantity は変更しない

3. Handy検品開始
   work_quantity = 8（デフォルト = shipped - received = 8 - 0）

4. 作業者が検品して完了
   received_quantity = 0 + 8 = 8
   status = PARTIAL（8 < 10）
```

### パターンB: 出荷実績なし → Handy検品

```
1. 発注確定
   expected_quantity = 10
   shipped_quantity  = null
   received_quantity = 0

2. Handy検品開始
   work_quantity = 10（デフォルト = expected - received = 10 - 0）

3. 作業者が検品して完了（実際は8個だった場合）
   作業者が work_quantity を 8 に修正
   received_quantity = 0 + 8 = 8
   status = PARTIAL
```

### パターンC: 出荷実績 → 管理画面で適用（Handy検品なし）

```
1. 発注確定
2. JX受信・照合 → shipped_quantity = 8
3. 管理画面「適用」ボタン → received_quantity = 8
```

### エンドポイント一覧

---

### POST /api/incoming/work-items/{id}/complete

作業完了（検品確定）

#### リクエスト

パラメータなし（作業データの `work_quantity` を使用）

#### サーバ側処理

```
total_received = schedule.received_quantity + work_item.work_quantity

if work_quantity >= remaining_quantity:
    → 入庫完了（CONFIRMED）
    → received_quantity = total_received
else:
    → 一部入庫（PARTIAL）
    → received_quantity += work_quantity
```

**重要**: 全量入庫時も `expected_quantity` ではなく実際のカウント（`received + work`）を使用。

## 端末側: 画面仕様

### 画面1: 入庫予定一覧

**API**: `GET /api/incoming/schedules?warehouse_id={id}`

**表示内容**:
- 商品画像、商品名、商品コード
- 容量・温度帯
- 数量表示（商品単位で集約）:

```
┌─────────────────────────────────┐
│ 商品A 720ml               常温  │
│ CD: 10001                       │
│ JAN: 4901234567890              │
│                                 │
│  予定: 10ケース                  │
│  出荷: 8ケース   ← shipped有時   │
│  済み: 0ケース                   │
│                                 │
│  → 検品待ち: 8ケース             │
└─────────────────────────────────┘
```

**「検品待ち」数量の決定ロジック（端末側）**:

```
if shipped_quantity != null:
    検品待ち = shipped_quantity - received_quantity
else:
    検品待ち = expected_quantity - received_quantity
```

**出荷実績がない場合の表示**:

```
┌─────────────────────────────────┐
│ 商品B 1800ml              常温  │
│ CD: 10002                       │
│                                 │
│  予定: 5ケース                   │
│  済み: 0ケース                   │
│                                 │
│  → 検品待ち: 5ケース             │
└─────────────────────────────────┘
```

- `shipped_quantity` が `null` の場合は「出荷」行を非表示。

```
│ [ キャンセル ]     [ 検品完了 ]  │
└─────────────────────────────────┘
```

**表示ルール**:

| 項目 | 表示条件 | データソース |
|------|---------|------------|
| 発注数 | 常に表示 | `schedule.expected_quantity` |
| 出荷実績 | `shipped_quantity != null` | `schedule.shipped_quantity` |
| 欠品 | `shortage_quantity > 0` | `schedule.shortage_quantity` |
| 入庫済み | `received_quantity > 0` | `schedule.received_quantity` |
| 検品数量（デフォルト） | 常に表示 | `work_quantity`（サーバ計算済み） |

**端末側の動作フロー**:

```
1. 一覧から商品をタップ → POST /api/incoming/work-items
2. レスポンスの work_quantity がデフォルト検品数量として表示
3. 作業者が数量を変更した場合 → PUT /api/incoming/work-items/{id}
4. 「検品完了」タップ → POST /api/incoming/work-items/{id}/complete
5. 完了後、一覧に戻る（一覧を再取得）
```

**API**: `GET /api/incoming/work-items?warehouse_id={id}&status=COMPLETED`

## 端末側に伝えるべき事項

### 1. 新規追加フィールド

以下のフィールドが全APIレスポンスに追加された:

| フィールド | 型 | 説明 |
|-----------|-----|------|
| `shipped_quantity` | int\|null | 出荷実績数。nullは未受信を意味する |
| `shortage_quantity` | int\|null | 欠品数（発注数 - 出荷数） |

**影響するAPI**: schedules一覧、schedules詳細、work-items の schedule オブジェクト内

### 2. 検品数量のデフォルト値が変わる

**旧**: 常に `expected_quantity - received_quantity`
**新**: 出荷実績がある場合は `shipped_quantity - received_quantity`

サーバが `work_quantity` として計算済みの値を返すので、端末側はそのまま使えばよい。

### 3. 表示の出し分け

- `shipped_quantity` が `null` → 出荷実績行を非表示
- `shipped_quantity` が値あり → 「出荷実績: X ケース」を表示
- `shortage_quantity > 0` → 「欠品: X ケース」を表示（警告色推奨）

### 4. 入力制約（端末側でのバリデーション推奨）

| 項目 | 制約 |
|------|------|
| 検品数量 | 0以上の整数。上限なし（超過入荷の可能性あり） |
| 入荷日 | 日付形式（デフォルト: 当日） |
| 賞味期限 | 日付形式 or null（デフォルト: 商品の default_expiration_days から計算） |
| ロケーション | `/api/incoming/locations` で検索可能 |

### 5. ステータスと操作可否

| schedule.status | 作業開始 | 数量修正 | 備考 |
|----------------|---------|---------|------|
| `PENDING` | ○ | - | 未入庫 |
| `PARTIAL` | ○ | - | 一部入庫済み、残りを検品可能 |
| `CONFIRMED` | × | △（履歴から） | 全量入庫済み |
| `TRANSMITTED` | × | × | 仕入連携済み、修正不可 |
| `CANCELLED` | × | × | キャンセル済み |
