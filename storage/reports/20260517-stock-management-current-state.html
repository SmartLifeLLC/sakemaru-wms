酒丸 在庫管理 現状整理

作成日: 2026-05-17 / 元資料: 2026-05-09 在庫整合性調査、2026-05-10 ローカル補正結果、現行 WMS 在庫同期実装

結論

現在の在庫問題は、単純な波動取消やピッキングリストの問題ではない。real_stocks と real_stock_lots の二重管理に対して、更新責務が一元化されていないことが根本原因である。

現時点の業務在庫の正は real_stocks と置く必要がある。一方、WMS の波動引当とピッキングは real_stock_lots とロケーションを見ている。このため、親在庫には数量があるのに、WMS が引当できるロットが存在せず欠品になる。

短期対応は real_stocks を再集計・上書きせず、波動生成対象に限って不足する引当可能ロットを補完すること。恒久対応は、売上・仕入・小売・倉庫移動・出荷確定・取消の全経路をロット同期サービスへ寄せ、在庫台帳を導入して差分理由を追跡可能にすることである。

WMS 波動で欠品になる直接原因

典型例は商品CD 156643。倉庫91の real_stocks.available_quantity は 24 だったが、有効な real_stock_lots が存在せず、波動では欠品になった。

WMS の引当候補は real_stock_lots と locations.available_quantity_flags で決まる。親在庫があっても、ACTIVE ロットがなく、かつ引当可能ロケーションに置かれていなければ、WMS は在庫なしと扱う。

Z-0-0 の扱い

調査時点の倉庫91デフォルトロケーション Z-0-0 は available_quantity_flags = 8 で、CASE/PIECE/CARTON のどれにも一致しなかった。そのため、ここにあるロットは引当対象外だった。

2026-05-10 のローカル補正では、ローカルDB上で Z-0-0 の floor_id と available_quantity_flags を変更して引当可能化し、安全対象のロットを作成した。ただしこれはローカル結果であり、本番適用時は本番の flags と運用意図を確認する必要がある。

現在の在庫同期機能

/admin/wave-settings の「在庫同期」は Warehouse91StockLotSyncService を呼び出す。対象は倉庫91限定で、real_stocks を正として ACTIVE ロット合計を同期する設計になっている。

推奨設計

短期: real_stocks 正のまま、波動対象だけ補完

  • 波動生成前、対象の warehouse_id / item_id だけを確認する。
  • parent_available - pickable_lot_available > 0 の場合だけ不足ロットを補完する。
  • real_stocks は更新しない。
  • 補完先は引当可能ロケーションに限定する。引当不可の Z-0-0 へ作らない。
  • 補完ログを残し、波動ID・商品・倉庫・数量・ロケーション・理由を追えるようにする。

中期: 出荷確定キューの契約統一

  • WMS キューが渡す earning_ids だけに依存せず、明細・数量・横持ち・欠品確定結果を読める形にする。
  • AI Core 側は WMS 予約または明細数量に基づいて、親とロットを同じ数量だけ減らす。
  • 自社セット構成品もキュー経由で減算できるようにする。

恒久: 在庫台帳とロット基準への移行

  • 在庫増減は台帳へ append し、現在庫は台帳またはロットから再現できるようにする。
  • real_stock_lots を実在庫の正、real_stocks を集計キャッシュへ段階移行する。
  • 外部在庫同期は親上書きではなく、差分調整ロットと台帳イベントで表現する。
  • 倉庫移動は出庫元ロットを減らし、入庫先に同属性ロットを作る。
  • POS / 小売取込は FEFO/FIFO でロット消費する。
