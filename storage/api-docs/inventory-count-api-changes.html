NEW JANコード辞書取得

GET /api/wms/inventory-counts/{id}/jan-codes

棚卸し対象の全商品について、JANコード → item_id の逆引き辞書を一括取得する。
初期同期時にローカル保存し、バーコードスキャン時の高速検索に使用。

レスポンス

{
  "jan_codes": {
    // 1つのJAN → 1商品（通常パターン）
    "4901234567890": [
      { "i": 100, "ct": "JAN", "t": "0", "q": 12 }
    ],

    // ケースJAN
    "14901234567897": [
      { "i": 100, "ct": "JAN", "t": "1", "q": 6 }
    ],

    // 1つのJAN → 複数商品（Android側で選択UI表示）
    "4902110352689": [
      { "i": 200, "ct": "JAN", "t": "0", "q": 12 },
      { "i": 301, "ct": "JAN", "t": "0", "q": 24 }
    ]
  }
}

フィールド定義

フィールド型説明
キー（JANコード）stringJANコード / ITFコード / 自社コード等
iinteger商品ID。items の item_id と一致
ctstringコード種別。例: JAN / OTHER
tstring0 = バラ / 1 = ケース / 2 = ボール / 9 = その他
qintegerケース入力時の換算入数。t=0 は items.capacity_case、t=1 はJANコード別入数
入数の扱い: バラJAN（t=0）を読んだ場合でも、表示とケース入力の換算には商品の items.capacity_case を使う。ケースJAN（t=1）はJANごとの入数を使う。
複数ヒット時の対応: 同一JANコードに複数の item_id が紐づく場合がある。Handy側で候補リストを表示し、ユーザーに選択させること。

CHANGE 棚卸し対象データ取得

GET /api/wms/inventory-counts/{id}/items

レスポンス変更箇所

旧 jan_codes（文字列配列）と own_codes（文字列配列）を廃止し、search_codes（オブジェクト配列）に統合。

Before (削除)
"jan_codes": ["4901234567890"]
"own_codes": ["10001"]
After (新規)
"search_codes": [
  { "c": "4901234567890", "ct": "JAN", "t": "0", "q": 12 },
  { "c": "14901234567897", "ct": "JAN", "t": "1", "q": 6 },
  { "c": "10001", "ct": "OTHER", "t": "0", "q": 12 }
]
フィールド型説明
search_codes[].cstringコード値
search_codes[].ctstringコード種別
search_codes[].tstring0 = バラ / 1 = ケース / 2 = ボール / 9 = その他
search_codes[].qintegerケース入力時の換算入数。バラJANは items.capacity_case、ケースJANはJANコード別入数
compact=1 の場合: search_codes は返されない（従来通り search_text にコード文字列を含む）。

CHANGE 商品スキャン検索

POST /api/wms/inventory-counts/{id}/scan

サーバー側の検索対象に item_search_information テーブル（JANコード）を追加。
13桁ゼロパディング正規化にも対応。

追加された検索対象

検索テーブルフィールド備考
item_search_informationsearch_stringJANコード完全一致
item_search_informationLPAD(search_string, 13, "0")13桁正規化一致
Handy側の対応: リクエスト・レスポンス形式に変更なし。サーバー側のみの改善のため、Handy側の修正は不要。
ただしレスポンスの search_codes フィールドは items API と同じ形式に変更（旧 jan_codes / own_codes → search_codes）。

Handy 側の対応事項まとめ

#対応内容優先度
1 初期同期時に GET /jan-codes を呼び出し、JANコード辞書をローカルDBに保存する 必須
2 バーコードスキャン時、まずローカルのJAN辞書から item_id を検索する 必須
3 同一JANで複数商品がヒットした場合の候補選択UI（商品名・商品CDを表示して選択） 必須
4 JAN辞書にないコードの場合、POST /scan にフォールバック 必須
5 旧 jan_codes / own_codes の参照を search_codes に移行 必須
6 ケースJANスキャン時、quantity_type: "CASE" を識別してケース数量入力欄に自動フォーカス（任意） 推奨
